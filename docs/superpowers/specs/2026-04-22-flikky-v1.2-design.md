# Flikky v1.2 设计文档 · 多会话批量导出 + UX 打磨

- 日期：2026-04-22
- 范围：v1.2「把多条历史会话一次性打包给电脑」+ B1 上传进度气泡 + B2 snackbar + WiFi 重绑
- 前置版本：v1.1（已发布，2026-04-21）
- 状态：待评审

## 1. 产品范围

### 1.1 In Scope（主干 = 多会话批量导出）

1. **主页右上角「导出」入口 → 多选模式**：TopAppBar action icon 进入 Selecting 状态，卡片左侧出现 checkbox，底部 action bar 显示「已选 N 个 / 开始导出」。
2. **进行中会话（`endedAt IS NULL`）不可选**：checkbox disabled，副标题注「结束服务后可导出」。
3. **导出服务 = `TransferService` 的 `ACTION_EXPORT` mode**：与 `ACTION_START` 互斥（同一时间只能一种）。试图在传输中启动导出 → APP 端提示「请先停止当前服务」。
4. **导出流式单 zip**：所有勾选会话一次性打包，浏览器端一个「下载全部 (X MB)」按钮触发。服务端流式 `ZipOutputStream` → 响应 `ByteWriteChannel`，不要求导出内容全部加载到内存。
5. **zip 内结构双文件格式**：每个会话子目录 `sessions/{id}_{safeName}/` 下同时给 `messages.txt`（人类可读）+ `messages.json`（机器可读）+ `files/`（原始二进制）。根目录额外一份 `README.txt` 记录导出元信息。
6. **zip 下载成功一次即自动停服**：`onZipSent` hook 调 `stopSelf()`；APP 端同时有手动「取消导出」按钮兜底。
7. **导出完成后给用户二择**：`DoneScreen` 显示「✅ 已导出 N 个会话」+ 两个按钮「保留本地」/「删除本地（将删除 N 条会话 + 约 X MB 文件）」。默认视觉突出「保留本地」，删除需视觉红色警告但不再二次确认（用户已有思考时间）。
8. **浏览器端导出页独立 `/export`**：复用 `/login` 做 PIN 认证；认证后 `GET /export` 列出所选会话清单（名称 / 消息数 / 文件数 / 总大小）+ 下载按钮。不重用 `/chat` 页。

### 1.2 In Scope（配套）

9. **B1 大文件上传进度气泡**：前端 `sendFile` 改用 `XMLHttpRequest` + `upload.onprogress`，上传开始立刻在本地 append 一条 status=`uploading` 占位泡；HTTP 响应返回后替换为正式泡；WS `file_added` 收到时按 `fileId` 去重。小文件（<1 MB）依然能有短暂但可见的泡。
10. **B2 mdui snackbar 替代 native alert**：新增 `showError(msg)` 全局 helper，统一走 mdui snackbar。替换范围：现有 `sendFile` 的 alert + 新增的 export 失败提示。登录页保持现状（已用 mdui）。
11. **WiFi 切换自动重绑服务器**：`TransferService` 注册 `ConnectivityManager.NetworkCallback`，`onCapabilitiesChanged` / `onLinkPropertiesChanged` 检测到新 IPv4 与当前绑定不同 → 停 Ktor → rebind new IP → 广播新 URL 给 ServingScreen。浏览器端旧 URL 自然断开，需用户主动重连（URL 会刷新显示）。

### 1.3 Out of Scope

- 会话 / 消息搜索（v1.3）
- 消息撤回（v1.3）
- 导出的反向操作：从 zip 导入回 APP（v1.3 或之后）
- HTTPS 自签证书（v2）
- 本地归档 at-rest encryption（v2）
- tee 文件异步化（backlog D12 可选优化；v1.2 接受 v1.1 现有同步 tee 行为）
- 设置页（FIFO 上限仍硬编码 20）
- 多浏览器同时下载同一 zip（一次性 PIN + 下载后即停，天然单次）

### 1.4 默认决定（无异议即生效）

- **zip 下载完即停**：与一次性 PIN 风格对齐，不做"多次下载留档"。用户要多次下载就再导一次。
- **会话子目录命名 = `{id}_{safeName}`**：id 保证唯一，name 辅助用户识别。非法字符（`/ \ : * ? " < > |` 等）替换为 `_`；超长名截断到 40 字符。
- **导出进程中 APP 崩溃 / 用户杀进程 = 失败**：浏览器端拿到的是半 zip；`stopService` 时没 `onZipSent` 回调过就不做"保留/删除"跳转，直接退回主页。
- **B1 占位泡失败处理**：上传失败时占位泡变红色 + 文案「上传失败（点击重试）」；点击重试时重新走同一路径。v1.2 先不做"断点续传"，失败即整个文件从头来。
- **WiFi 重绑提示位置**：ServingScreen 顶部加一个 banner「网络已切换，URL 已更新，浏览器需要重新打开新 URL」；浏览器端不做自动重连（URL 变了，无法识别）。

## 2. 架构与组件

### 2.1 目录结构（新增 / 改动）

```
com.example.flikky
├── export/                         [新增]
│   ├── ExportSession.kt            { sessionIds, pin, createdAt, consumed }
│   ├── ExportMode.kt               sealed: Idle | Armed(ExportSession) | Sending | Done
│   ├── ZipExporter.kt              流式打包核心（纯 Kotlin，无 Android / Ktor 依赖）
│   ├── MessagesTextFormatter.kt    Message -> messages.txt 行
│   └── MessagesJsonFormatter.kt    Message -> messages.json
├── server/
│   └── routes/
│       ├── ExportRoutes.kt         [新增] GET /export 页 + GET /api/export/zip
│       └── PinAuth.kt              [改]  导出 mode 下的 PIN 独立生命周期
├── service/
│   └── TransferService.kt          [改]  + ACTION_EXPORT + NetworkCallback 注册
├── session/
│   └── SessionState.kt             [改]  + exportMode: StateFlow<ExportMode>
├── network/
│   └── NetworkRebinder.kt          [新增] 监听 Wi-Fi 变化 + 计算新 IPv4
├── ui/
│   ├── home/
│   │   ├── HomeScreen.kt           [改]  selecting 状态 + 导出 icon + bottom action bar
│   │   └── HomeViewModel.kt        [改]  selection: Set<Long> / startExport()
│   ├── exporting/                  [新增]
│   │   ├── ExportingScreen.kt      URL + PIN + 取消 / Done 页 二择
│   │   └── ExportingViewModel.kt
│   └── serving/
│       └── ServingScreen.kt        [微调] WiFi 切换 banner
├── data/
│   └── SessionRepository.kt        [改]  + exportSnapshot(sessionIds): ExportSnapshot
└── FlikkyApp.kt                    [无改]
```

### 2.2 关键新组件职责

- **`ExportSession`**：一次性导出单元。PIN 单次使用（和 v1.0 一致），消费后丢弃。
- **`ExportMode` sealed class**：`Idle`（无导出在进行）/ `Armed(session)`（已启动，等浏览器连）/ `Sending(bytesSent)`（正在流输出 zip）/ `Done`（zip 已发完，等用户二择本地处理）。UI 订阅这个 Flow 决定渲染哪个屏。
- **`ZipExporter`**：纯 Kotlin，接收 `OutputStream` + `List<SessionSnapshot>`（不直接读 DB）→ 按规范写 zip。不做分块广播（流式就是分块）；不追加 log 字段（保持确定性，同输入同输出）。
- **`SessionRepository.exportSnapshot(ids)`**：查 DB 组装 `ExportSnapshot { sessions: List<SessionWithMessages>, files: File→metadata }`，不读文件内容（让 ZipExporter 再按需流式读）。
- **`NetworkRebinder`**：封装 `ConnectivityManager.NetworkCallback` + 当前绑定 IPv4 比较 + 触发「rebind 意图」事件。不直接碰 Ktor（`TransferService` 收到事件后决定做什么）。
- **`ExportRoutes`**：两条路由：
  - `GET /export`（需要 cookie）：返回 `assets/web/export.html`
  - `GET /api/export/zip`（需要 cookie）：`call.respondOutputStream { ZipExporter.write(it, snapshot) }`，完成后通过 callback 通知 `TransferService` stop。

### 2.3 互斥策略

- `TransferService.onStartCommand(ACTION_START)`：若 `exportMode != Idle` → 拒绝，发通知"导出中，无法启动传输"。
- `TransferService.onStartCommand(ACTION_EXPORT)`：若已有运行中的 TransferSession（`session.snapshot.value.serviceStartedAt != 0L` 且正常 start 路径）→ 拒绝，APP 端给 Toast「请先停止当前服务」。UI 层面 HomeViewModel.startExport() 启动前做同样检查并提示。
- `TransferService` 内用 `lateinit var mode: ServiceMode`（`Transfer` / `Export`），第一次 startCommand 定型；UI 无法改。

## 3. 关键流程

### 3.1 多会话导出流程

```
用户 A 主页点「导出」icon
  ↓ HomeViewModel: state = Selecting(selected=emptySet)
  ↓ HomeScreen: 渲染 checkbox；进行中会话 disabled
用户勾选 3 条 + 点「开始导出」
  ↓ HomeViewModel.startExport(ids):
  ↓   1. 检查是否有 transfer 在跑 → 若有，Toast + return
  ↓   2. repository.exportSnapshot(ids) 在 IO 协程组装
  ↓   3. ServiceLocator.session.armExport(snapshot, pin = PinGenerator.next())
  ↓   4. ctx.startForegroundService(Intent(ACTION_EXPORT))
  ↓   5. nav.navigate("exporting")
TransferService onStartCommand(ACTION_EXPORT):
  ↓ mode = Export; Ktor server 启动（仅 ExportRoutes + /login）
  ↓ 通知文案"正在提供导出"
ExportingScreen:
  ↓ 订阅 session.exportMode；显示 URL + PIN
浏览器打开 URL → PIN → /export 页 → 点击下载
  ↓ GET /api/export/zip:
  ↓   mode = Sending
  ↓   ZipExporter.write(outputStream, snapshot) 流式打包
  ↓   完成 → mode = Done → TransferService.stopSelf() (delayed 500ms)
ExportingScreen 转 Done 变体:
  ↓ 渲染「已导出」+ 保留 / 删除 两按钮
用户选「删除本地」:
  ↓ HomeViewModel.deleteSessions(ids); nav.popBackStack("home")
用户选「保留本地」:
  ↓ nav.popBackStack("home")
```

### 3.2 互斥失败路径

- 用户在 ServingScreen（传输中）想返回主页选择导出 → HomeScreen 是 popBackStack 到的，所以传输仍在跑；点「导出」进入 Selecting → 点「开始导出」→ HomeViewModel 检测到 transfer running → Toast 「请先停止当前服务」。Selecting 状态保持，不回主页。
- 导出中想启动传输 → 导出期间 ServingScreen 不可达（主页被 ExportingScreen 盖住）。不需要额外保护。

### 3.3 B1 上传进度流程

```
用户点上传文件 → 浏览器 app.js:
  ↓ const placeholderId = `local-${Date.now()}`
  ↓ renderUploadingBubble({ localId, name, total: file.size })
  ↓ const xhr = new XMLHttpRequest()
  ↓ xhr.upload.onprogress = (e) => updateBubbleProgress(localId, e.loaded / e.total)
  ↓ xhr.onload = () => {
  ↓   if (xhr.status === 200) {
  ↓     const dto = JSON.parse(xhr.responseText)
  ↓     markBubbleCompleted(localId, dto.fileId)  // 关联远端 fileId
  ↓   } else showError(`上传失败 (${xhr.status}): ${file.name}`)
  ↓ }
  ↓ xhr.onerror = () => showError(`上传出错: ${file.name}`)

WS onEvent(file_added):
  ↓ if (document.querySelector(`[data-file-id="${ev.fileId}"]`)) return  // dedup
  ↓ renderCompletedBubble(ev)
```

关键点：服务端 `POST /api/files` 当前返回 JSON `{ fileId, name, sizeBytes, ... }`；前端拿回后在本地把"uploading 占位泡"的 data-file-id 设为服务端 fileId。WS 广播到达时，查询到已有同 fileId 节点，跳过。小文件（< 1 MB）从 onprogress 到 onload 之间可能只有一帧，占位泡闪一下但不会消失——可接受。

### 3.4 WiFi 重绑流程

```
TransferService onStartCommand(ACTION_START):
  ↓ serverIp = NetworkInfo.wifiIpv4() or throw
  ↓ startKtor(serverIp)
  ↓ connectivityManager.registerNetworkCallback(networkRequest, callback)

callback.onLinkPropertiesChanged(network, linkProps):
  ↓ newIp = linkProps.linkAddresses.firstIpv4()
  ↓ if (newIp != null && newIp != serverIp) {
  ↓   stopKtor()
  ↓   serverIp = newIp
  ↓   startKtor(serverIp)
  ↓   session.updateServerUrl("http://$newIp:$port")
  ↓   notifyUi("网络已切换，URL 已刷新")
  ↓ }

callback.onLost(network):
  ↓ // 不主动停服；等新 network 上来或用户手动停。
  ↓ session.markNetworkLost()
```

UI 层面，ServingScreen 订阅 `session.snapshot.map { it.networkStatus }`：
- `Ok` → 正常显示 URL
- `Switching` → 顶部 banner「正在切换网络…」（Ktor 停→起之间的短暂状态，< 1s）
- `Lost` → banner「已失联，等待网络恢复」+ URL 置灰
- `Switched` → banner「网络已切换，URL 已更新为 ...，请让浏览器重新打开」+ 一个 "我知道了" 按钮

## 4. 数据格式规范

### 4.1 zip 结构

```
flikky-export-20260425-143022.zip
├── README.txt
└── sessions/
    ├── 12_04-22 14·30 与小明/
    │   ├── messages.txt
    │   ├── messages.json
    │   └── files/
    │       ├── doc.pdf
    │       ├── photo.jpg
    │       └── ...（原始文件名；UUID fileId 不保留）
    └── 13_04-22 18·00/
        ├── messages.txt
        ├── messages.json
        └── files/
            └── report.docx
```

**命名规则：**
- 会话目录：`{id}_{safeName}`，`safeName = name.replace("/\\:*?\"<>|", "_").take(40)`（`:` 原本合法但部分 FS 不喜欢，用 `·` U+00B7 替代以保留视觉分隔感）
- `files/` 下用原始文件名 `Message.File.name`；重名时自动后缀 `_2`, `_3`（按 timestamp 顺序递增）

### 4.2 README.txt（zip 根）

```
Flikky Export
Version: 1.2
Exported at: 2026-04-25 14:30:22 (+0800)
Sessions: 3
Total messages: 42
Total files: 7
Total bytes: 15728640

此 zip 包含你从 Flikky 手机端勾选导出的 N 条会话。
每个会话一个子目录，内含 messages.txt（纯文本消息日志）、messages.json（结构化数据）和 files/（原始附件）。
```

### 4.3 messages.txt

```
# 会话名: 04-22 14:30 与小明
# sessionId: 12
# 起止: 2026-04-22 14:30:15 ~ 2026-04-22 15:02:18
# 消息: 18 条, 文件: 3 个

[2026-04-22 14:30:15] [PHONE  ] hello world
[2026-04-22 14:31:02] [BROWSER] 收到，我看看
[2026-04-22 14:35:40] [BROWSER] [文件] doc.pdf (1.2 MB)
[2026-04-22 14:40:11] [PHONE  ] 好的
...
```

### 4.4 messages.json

```json
{
  "sessionId": 12,
  "name": "04-22 14:30 与小明",
  "startedAt": 1713768615000,
  "endedAt": 1713770538000,
  "pinned": false,
  "messages": [
    {
      "type": "text",
      "ts": 1713768615000,
      "origin": "PHONE",
      "content": "hello world"
    },
    {
      "type": "file",
      "ts": 1713768940000,
      "origin": "BROWSER",
      "fileId": "a45706d9-a576-42c5-bfd4-de3a06743103",
      "name": "doc.pdf",
      "mime": "application/pdf",
      "sizeBytes": 1234567,
      "relativePath": "files/doc.pdf"
    }

```

`fileId` 保留是为了未来"导入回 APP"功能对齐；当前导出侧用不到。`relativePath` 指向 zip 内路径（含重名后缀）。

## 5. 测试策略

### 5.1 纯 JVM 单测

- **`ZipExporterTest`**（无 Robolectric，纯 Kotlin）：给定 mock `ExportSnapshot` + 模拟文件目录 → 写到 `ByteArrayOutputStream` → 用 `ZipInputStream` 解回来验证目录结构 / 文件名 / 文本内容。
- **`MessagesTextFormatterTest`**：各种 Message 类型（Text / File / 混合）→ 期望 txt 输出；时间戳格式化；文件名含换行等边界字符。
- **`MessagesJsonFormatterTest`**：JSON schema 严格对照；messages 顺序 = ascending timestamp；timestamp 为毫秒整数不是字符串。
- **`ExportRoutesTest`**（Ktor test-host）：`GET /export` 未登录 → 401 / 重定向到 /login；登录后 `GET /api/export/zip` 返回 200 + `Content-Type: application/zip` + `Content-Disposition`。
- **`HomeViewModelSelectingTest`**：进入 / 退出 Selecting 状态；勾选 / 取消勾选；`startExport` 在 transfer 运行时拒绝。
- **`NetworkRebinderTest`**：给定 linkProps 事件序列，期望正确发出 rebind / ignore 信号（逻辑纯函数）。

### 5.2 仪器测试（`androidTest/`）

- **`TransferServiceModeTest`**：ACTION_START → running → ACTION_EXPORT 拒绝；stop → ACTION_EXPORT 接受。
- **`ExportEndToEndTest`**（可选，若设备稳定）：在设备上启 service → mock HTTP 客户端发 PIN 登录 → GET zip → 用 `ZipInputStream` 解验内容。

### 5.3 装机冒烟（人工验证清单）

- 主页右上角有「导出」icon；点击进入多选模式
- 进行中会话灰掉；非进行中可勾
- 勾 0 条时「开始导出」按钮 disabled
- 导出服务运行时通知栏文案区分于正常传输
- zip 下载后 APP 跳 Done 屏；保留 / 删除两按钮分别生效
- zip 结构正确（手工打开验）
- 大文件上传看得到进度；失败能看到 snackbar
- WiFi 切换后 APP 显示 banner + 新 URL；浏览器打开新 URL 能连上

## 6. 决策（进入 plan 前冻结）

- **D18 · 导出服务 = ACTION_EXPORT mode（复用 TransferService）**：换新 Service 会多一套通知 / PIN / 端口维护；v1.x 场景下用户只会做一件事。互斥是功能需求，不是限制。
- **D19 · 流式单 zip（不多 zip 不 JSON-only）**：用户一次点击得到一份可存档的归档；messages.txt + messages.json 双格式兼顾人看与机器读；未来 v1.3 导入回 APP 直接用 messages.json。
- **D20 · 导出完成后给"保留 / 删除"两按钮**：用户反馈明示"今后自己选"；选择项放在完成页而不是勾选时，避免误操作。默认视觉突出「保留本地」，降低误删风险。
- **D21 · B1 前端乐观 UI + XHR progress**：不改服务端（服务端改动会引入多客户端 WS 流量放大 + dedup 复杂度）；XHR 是浏览器原生唯一能拿 upload progress 的 API；dedup key = 服务端返回的 fileId。
- **D22 · WiFi 切换 → 停旧 Ktor + 起新 Ktor，不做连接迁移**：Ktor CIO 没有 rebind 能力，必须完整重启；旧浏览器连接必然断，用户需重新打开。尝试"保留 WS"复杂度远超收益。
- **D23 · 导出期间不做「暂停传输 → 导出 → 恢复」的状态保存**：v1.2 用户一次只做一件；要"一边传一边导"请用 v1.3。

## 7. 验收标准

- `./gradlew assembleDebug testDebugUnitTest` 全绿
- 新增 JVM 单测覆盖：ZipExporter / Formatters / ExportRoutes / HomeViewModel selecting / NetworkRebinder
- 装机冒烟清单（§5.3）全通过
- 导出产出的 zip 用系统归档工具能打开，messages.txt 人类可读，messages.json 能被 `jq . messages.json` 正确解析
- 大文件（>50 MB）上传过程中进度气泡实时刷新，误差 < 5% 总大小
- 切换 WiFi 后 APP 顶部 banner 出现，新 URL 正确显示；浏览器打开新 URL 登录能恢复导出

## 8. 不确定项（plan 阶段决定）

- ZipOutputStream 使用 `java.util.zip` 还是 `kotlinx.io`？前者 JVM 内置，够用；后者与 Ktor 协程 IO 更贴，但依赖大。倾向前者。
- `NetworkCallback` 在 Android 7+ 可用，Flikky minSdk=33 没问题，但在锁屏 / Doze 下的行为需要仪器测试确认。
- 导出完成后的"保留/删除"两个按钮，点击删除是否走和主页长按删除一致的 `repository.deleteSession` 路径？倾向是。
