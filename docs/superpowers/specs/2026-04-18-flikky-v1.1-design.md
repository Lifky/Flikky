- 日期：2026-04-18
- 范围：v1.1「完整体验：持久化 + 列表 + 重命名 / 置顶 / 删除」
- 前置版本：v1.0（tag 已打，test3 全通过）

## 1. 产品范围

### 1.1 In Scope

1. **会话持久化**：消息 + 文件二进制都落本地磁盘；进程被杀或设备重启后重开 APP 仍可回看。
2. **主页 = 会话列表**：代替 v1 的极简启动页。列表项含会话名、起止时间、副标题（预览文本 / 消息-文件-字节统计）；空态有引导卡片。
3. **单会话查看**：点击会话进入 `HistoryScreen`，复用 v1 的 `MessageBubble` 渲染消息，文件可通过 FileProvider 重新打开（同 v1）。
4. **会话管理**：重命名、删除、置顶——长按列表项或 overflow 菜单触发。
5. **FIFO 淘汰**：自动保留最近 **20 条**非置顶会话；新会话开始时扫一遍，多出的从最旧起整条 + 文件目录一起删。置顶会话不占这 20 的配额。
6. **空会话回滚**：一次服务启停期间没有任何消息与文件 → 停服时删除这条会话记录 + 文件目录，不在列表里留痕。
7. **实时写盘**：消息/文件元数据一 append 就落 Room；进程被系统杀死不会丢已发送内容。
8. **手机推送文件同步 tee**：选中 content:// URI 后同步拷贝到 `sessions/{id}/files/` 作为持久副本，浏览器下载从副本读。代价是大文件选中到可下载之间多一次拷贝延迟。
9. **孤儿恢复**：APP 启动时扫描 `endedAt IS NULL` 的会话 → 补 `endedAt` 或按空会话规则删除；同时清理 DB 中已不存在 session 的孤儿文件目录。

### 1.2 Out of Scope

- 会话 / 消息搜索（v1.2 或 v1.3）
- 选择多个会话批量导出到电脑（v1.2，一次性临时服务模式）
- 消息撤回（v1.3）
- 本地加密（v2）
- 设置页（v1.1 不做——保留上限 20 为硬编码，用户不可改；需要调整时改代码发版）
- v1 → v1.1 数据迁移（v1 本身不存数据，直接建新 DB）

### 1.3 默认决定（无异议即生效）

- **tee 文件拷贝**：采纳（v1.1 阶段接受大文件选中延迟，换取离线回看）
- **空会话回滚位置**：放在 `endSession` 里（不是"开始时不 insert"）——简化并发路径
- **previewText 冗余**：保留——列表高频、消息表低频更新，空间换时间。若将来列表需求变化可去掉。

### 2.2 关键新组件职责

- **`FlikkyDatabase`**：Room 单例，唯一进程级 DB 句柄。
- **`SessionRepository`**：业务入口。所有 UI 和 Service 对持久层的访问都经过它，不直接碰 DAO。仅依赖 `dao + filesDir: File + now: () -> Long`；不持有 Context（Room 构建在上层 `ServiceLocator` 里完成）。
- **`SessionFileStore`**：替代 v1 `AndroidFileStore`。给定 sessionId 返回该会话的 `files/` 目录；新增 `archivePhoneFile(sessionId, uri): File` 做 tee；实现 `server/routes/FileStore` 接口。
- **`HistoryScreen / ViewModel`**：只读回看，复用 `MessageBubble` 与 v1 的 `ViewModel.openFile` 逻辑（路径换成 sessions/{id}/files/）。

## 3. 数据模型

### 3.1 Entities

```kotlin
@Entity(tableName = "sessions")
data class SessionEntity(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    val startedAt: Long,
    val endedAt: Long?,          // null = 进行中
    val name: String,            // 默认 "MM-DD HH:mm 会话"；用户可改；长度 ≤ 40
    val pinned: Boolean = false,
    val messageCount: Int = 0,
    val fileCount: Int = 0,
    val totalBytes: Long = 0,
    val previewText: String? = null, // 首条文本消息前 40 字符
)

@Entity(
    tableName = "messages",
    foreignKeys = [ForeignKey(
        entity = SessionEntity::class,
        parentColumns = ["id"], childColumns = ["sessionId"],
        onDelete = ForeignKey.CASCADE,
    )],
    indices = [Index("sessionId"), Index("timestamp")],
)
data class MessageEntity(
    @PrimaryKey val id: Long,       // 沿用 IdGen.newMessageId()
    val sessionId: Long,
    val origin: String,             // "PHONE" | "BROWSER"
    val timestamp: Long,
    val kind: String,               // "TEXT" | "FILE"
    val content: String? = null,    // TEXT-only
    val fileId: String? = null,     // FILE-only
    val fileName: String? = null,
    val fileSize: Long? = null,
    val fileMime: String? = null,
    val fileStatus: String? = null, // OFFERED | IN_PROGRESS | COMPLETED | FAILED
)
```

**单表 + `kind` 判别 vs 两张表：** 选单表，理由——历史页要按 timestamp 混排，单表一次查询即得；两表需要 UNION 或内存合并。null 字段较多是代价。

### 3.2 DAO 关键查询

```kotlin
@Dao interface SessionDao {
    @Insert fun insert(s: SessionEntity): Long
    @Update fun update(s: SessionEntity)
    @Delete fun delete(s: SessionEntity)
    @Query("SELECT * FROM sessions WHERE id = :id") fun getById(id: Long): SessionEntity?
    @Query("SELECT * FROM sessions ORDER BY pinned DESC, startedAt DESC")
    fun observeAll(): Flow<List<SessionEntity>>
    @Query("""SELECT * FROM sessions
              WHERE pinned = 0 AND endedAt IS NOT NULL
              ORDER BY startedAt ASC""")
    fun nonPinnedOldestFirst(): List<SessionEntity>
    @Query("SELECT * FROM sessions WHERE endedAt IS NULL")
    fun listUnfinished(): List<SessionEntity>
}

@Dao interface MessageDao {
    @Insert fun insert(m: MessageEntity)
    @Update fun update(m: MessageEntity)
    @Query("SELECT * FROM messages WHERE sessionId = :sid ORDER BY timestamp ASC")
    fun observeBySession(sid: Long): Flow<List<MessageEntity>>
    @Query("SELECT * FROM messages WHERE sessionId = :sid ORDER BY timestamp ASC")
    fun listBySession(sid: Long): List<MessageEntity>
    @Query("SELECT COUNT(*) FROM messages WHERE sessionId = :sid") fun countBySession(sid: Long): Int
    @Query("""SELECT content FROM messages
              WHERE sessionId = :sid AND kind = 'TEXT'
              ORDER BY timestamp ASC LIMIT 1""")
    fun firstTextContent(sid: Long): String?
}
```

### 3.3 Repository 对外方法

```kotlin
class SessionRepository {
    // 写
    suspend fun beginSession(name: String, startedAt: Long): Long
    suspend fun appendMessage(sessionId: Long, msg: Message)
    suspend fun endSession(sessionId: Long, endedAt: Long)   // 内部聚合统计 + 空会话回滚
    suspend fun fifoSweep()                                   // 结束态 + 非置顶 超 20 条 → 删最旧
    suspend fun finalizeOrphans()                             // APP 启动时调

    // 会话管理
    suspend fun rename(sessionId: Long, newName: String)      // trim + 空名回默认 + 40 字截断
    suspend fun setPinned(sessionId: Long, pinned: Boolean)
    suspend fun deleteSession(sessionId: Long)                // DB 行 + 文件目录

    // 查
    fun observeSessions(): Flow<List<SessionEntity>>
    fun observeSession(sessionId: Long): Flow<SessionEntity?>
    fun observeMessages(sessionId: Long): Flow<List<Message>>
}
```

### 3.4 Message ↔ Entity mapper

在 Repository 层做转换。UI 与 Server 层继续用 `session.Message`（对 v1 代码侵入最小）。

### 3.5 FIFO 触发点

每次 `beginSession` 开始时调一次 `fifoSweep`：
- `nonPinnedOldestFirst()` 拿结束态、非置顶会话列表。
- 若 `size > 20`，从头（最旧）取 `size - 20` 条，逐个 `delete(session)`（CASCADE 会带走 messages），再 `File(filesDir, "sessions/${id}").deleteRecursively()`。

选"开始时"而非"结束时"：避免本次刚结束的会话立即被自己的 FIFO 判定误伤的极端情况。

## 4. 数据流

### 4.1 启动服务

```
HomeScreen FAB 点击
  → HomeViewModel.startService()
  → Intent(ACTION_START) → TransferService.startTransfer()
      ├─ name = formatDefaultName(now())      "04-18 14:30 会话"
      ├─ sessionId = repo.beginSession(name, startedAt = now())
      ├─ SessionState.reset(sessionId = sessionId)
      ├─ repo.fifoSweep()
      ├─ KtorServer.start() (同 v1)
      └─ startForeground(...)
  → Nav 到 ServingScreen
```

### 4.2 收发消息（实时写盘）

BROWSER 侧（`MessageRoutes` POST /api/messages）与 PHONE 侧（`TransferController.sendText`）在 add 到 `SessionState` 的同时调用 `repo.appendMessage(sessionId, msg)`。appendMessage 是 suspend，走 IO 调度器；失败时 catch + Log + Toast（由上层决定是否 Toast），**不阻塞**消息广播。

### 4.3 文件上传（BROWSER → 手机）

`FileRoutes` 现在从 `fileStore.fileDir(sessionId)` 取目标目录。写入完成后与 v1 一样构造 `Message.File(COMPLETED)` → `session.addMessage` → `repo.appendMessage`。

### 4.4 文件推送（手机 → BROWSER）

`TransferController.offerFile(uri, resolver)` 的 v1 实现是注册一次性 InputStream 给 FileRoutes 的 phone-push 分支。v1.1 改为：
1. `fileStore.archivePhoneFile(sessionId, uri, fileId)` 同步拷贝到 `sessions/{id}/files/{fileId}`；
2. 构造 `Message.File(COMPLETED)` 直接 append；
3. `FileRoutes GET /api/files/{id}` 的 phone-push 分支改为：若文件在当前会话目录里存在则直接 respond 该文件；否则 fallback 到"上传文件"分支（同目录；语义统一）。

pending map 机制（v1 的 `PushedFile`）**废弃**。

### 4.5 停止服务

```
TransferService (ACTION_STOP | onDestroy)
  → stats = session 内消息 / 文件的聚合
  → repo.endSession(sessionId, endedAt = now(), stats)
      ├─ 若 messageCount == 0 && fileCount == 0:
      │     delete(sessionRow) + deleteRecursively(sessions/{id})
      └─ else: update(row with endedAt, counts, previewText = firstTextContent(sid)?.take(40))
  → ktor.stop; stopForeground; stopSelf
```

## 5. 错误与边界

- **Room 写失败（磁盘满 / 权限异常）**：appendMessage catch + Log + Toast "历史保存失败"，不阻塞消息广播。失败消息不做补写。
- **beginSession 失败**：直接停服 + Toast。没有 sessionId 后续数据流走不通，必须中止。
- **文件 tee 失败**：`archivePhoneFile` 抛异常 → `offerFile` catch → Toast "文件无法读取"；不 append 消息，不 broadcast。
- **进程被杀 / 设备重启**：下次 APP 启动时 `FlikkyApp.onCreate` 触发 `repo.finalizeOrphans()`，扫 `endedAt IS NULL` 的会话：有消息/文件的补 `endedAt = max(lastMessageTimestamp, startedAt)` + 重算统计；零消息零文件的按"空会话"删除。同步清理 `sessions/` 下 DB 里不存在的 session id 目录。
- **FIFO 文件目录删失败**：DB 已 CASCADE 删消息，文件目录 `deleteRecursively` 返 false 只记 Log——孤儿文件下次 finalize 时还会再清。
- **正在进行中的会话**：列表项菜单里 rename / delete / pin 禁用（tooltip"进行中不可操作"）；HistoryScreen 进入后 TopAppBar overflow 菜单同样禁用；FIFO query 已过滤 `endedAt IS NOT NULL`。
- **重命名边界**：trim 首尾空格；空名退回默认；超 40 字符截断。
- **v1 → v1.1 数据迁移**：无。v1 没数据表；v1.1 首次跑直接建。`exportSchema = false`（v1.1 简化）；v1.2 起改 true 并导 schema 到 `schemas/`。

## 7. 设计决策记录（便于日后迭代时定位）

以下决策将在实施完成后整理进 `docs/notes/decisions.md`，这里先汇总：

- **D10 · Room 作为 v1.1 持久层**：相较 JSON 文件方案，Room 为 v1.3 搜索 / 撤销铺路；代价是 KSP + schema 管理。确定由用户拍板（2026-04-18 brainstorming）。
- **D11 · 文件按 sessionId 分目录**：`filesDir/sessions/{id}/files/`。FIFO 与后续 v1.2 导出时单目录即整包，天然对位。
- **D12 · 手机推送文件同步 tee**：持久副本换离线回看；大文件选中有一次性拷贝延迟。v1.1 接受；v1.2 可考虑"先响应浏览器流，后台异步 tee"优化。
- **D13 · SessionEntity 冗余字段**（messageCount / fileCount / totalBytes / previewText）：列表高频展示，避免每次扫 messages 表。update 时机集中在 endSession（以及重命名不动这些）。如果将来列表字段频繁变化可去掉转为 view 查询。
- **D14 · 单表 messages + kind 判别**：历史页按 timestamp 混排，单表查询自然；代价是 null 字段。两表合并的开销不值得。
- **D15 · FIFO 触发点 = beginSession 开始**：避免本次刚结束的会话被自己的判定误伤的极端情况（如配额正好卡在 20 上）。
- **D16 · 空会话回滚位置 = endSession**：在结束时统一判定是否回滚，不干扰消息并发路径。对比"延迟 insert 会话行"的方案，并发更简单。
- **D17 · v1.1 保留上限硬编码 20**：设置页延后；未来若需要用户可配置则加设置页 + 读 SharedPreferences。
