## M1. Export 基础设施（纯 Kotlin，零 Android / Ktor 依赖）

**依赖：** 无。可独立派 subagent，4 个任务并行。

- **T1 · `export/ExportSession.kt` + `ExportMode.kt`**：数据类 + sealed class 定义；无业务逻辑，仅描述状态。验证：编译通过 + smoke unit test 构造各状态。
- **T2 · `export/MessagesTextFormatter.kt`**：`format(session: SessionSnapshot): String`。含 header（会话名/起止/统计）+ 每行 `[ts] [origin] ...`。验证：`MessagesTextFormatterTest` 覆盖 Text / File / 空会话 / 含换行 / 含特殊字符。
- **T3 · `export/MessagesJsonFormatter.kt`**：生成符合 spec §4.4 的 JSON。用 `kotlinx.serialization` 还是 `org.json`？倾向前者（项目里已用）。验证：`MessagesJsonFormatterTest` 严格字段断言 + `JsonElement` 解析回来校验。
- **T4 · `export/ZipExporter.kt`**：`write(out: OutputStream, snapshot: ExportSnapshot)`。内部用 `java.util.zip.ZipOutputStream`，会话目录 + messages.txt/json + files/，根 `README.txt`。纯函数，时间戳作为入参。验证：`ZipExporterTest` 用 `ByteArrayOutputStream` + `ZipInputStream` 解回 entries 全字段断言。

## M2. Repository 与 Session 状态扩展

**依赖：** M1（需要 ExportSnapshot / ExportMode 类型）

- **T5 · `SessionRepository.exportSnapshot(ids: List<Long>): ExportSnapshot`**：从 DB 组装；不读文件内容（让 ZipExporter 按需流式读）。验证：`SessionRepositoryExportSnapshotTest`（Robolectric + in-memory DB），覆盖空 ids / 含进行中会话（应过滤掉）/ 含不存在 id（应静默忽略）。
- **T6 · `SessionState.exportMode: StateFlow<ExportMode>` + `armExport / consumeExport / clearExport`**：内存 state machine，不碰 DB。验证：`SessionStateExportModeTest` 状态迁移。

## M3. Ktor 导出路由

**依赖：** M1（ZipExporter）+ M2（SessionState.exportMode）

- **T7 · `server/routes/ExportRoutes.kt`**：`GET /export`（返回 export.html）+ `GET /api/export/zip`（`respondOutputStream { ZipExporter.write(...) }`，完成后 mode → Done 并触发 stopSelf）。需 cookie 鉴权。验证：`ExportRoutesTest`（ktor test-host），未登录 401 / 登录后 zip 200 + header 正确。
- **T8 · `assets/web/export.html` + `export.js` + CSS**：会话清单 + 大下载按钮；前端不写复杂逻辑，fetch blob 走浏览器默认下载。验证：代码审阅 + 装机冒烟。
- **T9 · `KtorServer` 按 `ServiceMode` 决定挂 transfer 路由还是 export 路由**：一份 Ktor server，路由安装时机用 lazy delegate 或 conditional module。验证：`KtorServerTest` mode=Export 时 `/api/messages` 返回 404。

## M4. TransferService 多 mode

**依赖：** M3（ExportRoutes 挂得上 Ktor）

  - 通知文案："正在提供导出（N 个会话 / X MB）"，PendingIntent 跳 ExportingScreen

## 风险与减缓

- **Ktor `respondOutputStream` 的背压**：流式 zip 写的太快浏览器跟不上会内存堆积。v1.2 接受（用户一次导一两 GB 也在正常范围），真压力大再改 chunked。
- **`NetworkCallback` 在 Doze / 锁屏下**：可能错过回调。v1.2 接受；v1.3 再看是否要加轮询兜底。
- **ZipOutputStream 的 UTF-8 文件名**：Java 默认 CP437，需要 `ZipOutputStream.setDefaultEncoding("UTF-8")` 或在 constructor 显式传。Windows 解压某些工具有坑，`README.txt` 里说明一下。
- **B1 dedup key = fileId，如果上传并发**：两个 XHR 同时跑都要占位泡；每个有自己的 localId，服务端返回 fileId 后独立关联。不共享状态。
- **互斥但用户强行杀 APP 后的恢复**：Service 被系统重建时 mode 状态丢；依赖 `SessionState.exportMode` 是内存 state，重启即 Idle，安全。但要验证仪器测试。
